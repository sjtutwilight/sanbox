## Nacos 实验参数动态配置

- `load-executor` 已接入 Nacos Config，可以通过 `experiment.dynamic-config.*` 配置打开（或设置环境变量 `EXPERIMENT_DYNAMIC_CONFIG_ENABLED=true`、`NACOS_SERVER_ADDR=127.0.0.1:8848` 等）。
- 默认读取 `group=EXPERIMENT`、`dataId=load-executor-experiment-params.json` 的 JSON 文件；修改内容后 Nacos 会推送给压测节点，新的参数会在数秒内生效。
- JSON 结构为一个 Map，key 形如 `experimentId/groupId/operationId`，value 为需要覆盖到 payload 上的参数。例如：

```json
{
  "*/*/*": {
    "sequence": 0
  },
  "favorite/default/read_cache_aside": {
    "ttlSeconds": 45,
    "userCount": 2000
  },
  "kafka_kline/market-data/*": {
    "interval": "5s",
    "symbols": "BTCUSDT:70,ETHUSDT:30"
  }
}
```

- 覆盖顺序为：`*/*/*`（全局） → `experimentId/*/*` → `experimentId/groupId/*` → `experimentId/groupId/operationId`，后者会覆盖前者的同名字段。
- 被覆盖的参数会合并进 `Command.dataRequest`、payload 模板的输出，再传给具体实验操作，因此可以在实验运行过程中动态调整 TTL、Kafka topic 等控制面参数去观测指标变化。
- 现在可以通过提交实验命令时的 `overrides` 部分直接写入 Nacos（`experimentId/groupId/operationId` 对应节点会被覆盖），方便把 UI 中的调参一键同步到动态配置；若 `overrides` 为空则不会修改 Nacos。
- 若 Nacos 中的 dataId 为空，则回退到静态配置，不会影响原有行为。

## 实验：ConcurrentHashMap 调优（元数据管理 / 统一查询服务）

> 目的：找出在高并发读多写少的“元数据管理”和“统一查询服务”场景下，`ConcurrentHashMap` 的合适尺寸、竞争度与装填因子配置，验证 map 访问热点、重建成本对延迟和 CPU 的影响，并形成调参基线。

### 一、前置环境

- 启动依赖：`docker-compose up -d nacos mysql redis kafka grafana prometheus`（本仓库 compose 已包含 nacos，无需额外配置）。
- 服务：`./run-control-plane.sh`、`./run-load-executor.sh`；确保 `EXPERIMENT_DYNAMIC_CONFIG_ENABLED=true`、`NACOS_SERVER_ADDR=127.0.0.1:8848`。
- Nacos：`group=EXPERIMENT`、`dataId=load-executor-experiment-params.json`；前端可在提交命令时通过 `overrides` 同步写入。
- 监控：Grafana（JVM/Redis/MySQL dashboard）、async-profiler（CPU/锁）、JFR（对象分配）、应用日志（命中率、重建次数）。

### 二、场景映射与压测模型

- **元数据管理（experimentId=`metadata_cache`, groupId=`default`）**
  - 读：`read_catalog`（查询库/表/列/分区属性，命中 map 返回；miss 触发重建）
  - 写/失效：`invalidate_catalog`（模拟 schema 发布或权限变更，失效缓存）
  - 典型比：读写 100:1 或 1000:1；热点 key 占比 60%+。
- **统一查询服务 UDQS（experimentId=`udqs_plan`, groupId=`default`）**
  - 读：`lookup_plan`（按 `queryTemplateId+tenant` 查路由/计划）
  - 写/刷新：`refresh_plan`（模拟统计信息更新、分片调整导致的计划重建）
  - 典型比：读写 200:1；热点模板 5% key 占 70% 流量。

> 若尚未实现上述 operation，可按 `FavoriteExperimentGroup` 模板添加 ExperimentGroup，Invoker 中将热点 key 映射到 `ConcurrentHashMap`，并暴露下面的参数。

### 三、Nacos 参数模板（可在前端实时调节）

```json
{
  "*/*/*": {
    "initialCapacity": 1024,
    "loadFactor": 0.75,
    "maxEntries": 200000,
    "valueSizeBytes": 2048,
    "computeCostMicros": 200,
    "hotKeyRatio": 0.7,
    "hotKeyCount": 200,
    "keySpaceSize": 100000
  },
  "metadata_cache/default/read_catalog": {
    "hotKeyRatio": 0.85,
    "computeCostMicros": 400,
    "ttlSeconds": 60
  },
  "metadata_cache/default/invalidate_catalog": {
    "invalidateRange": 500,
    "burstIntervalMillis": 30000
  },
  "udqs_plan/default/lookup_plan": {
    "hotKeyRatio": 0.7,
    "computeCostMicros": 250,
    "ttlSeconds": 20
  },
  "udqs_plan/default/refresh_plan": {
    "burstIntervalMillis": 60000,
    "refreshBatch": 200
  }
}
```

- `initialCapacity/loadFactor/maxEntries`：控制 map 装载与扩容，观察 rehash/尾延迟。
- `valueSizeBytes/computeCostMicros`：模拟缓存重建成本；提高可放大锁竞争。
- `hotKeyRatio/hotKeyCount/keySpaceSize`：调节热点分布，引导到常见的写少读多热点。
- `burstIntervalMillis/refreshBatch`：控制写/失效风暴频率，观察 map 退化。

### 新增实验 operation（前端直接可选）

- experimentId=`metadata_cache`，groupId=`default`
  - `read_catalog`（CONTINUOUS_READ）：热点读 + computeIfAbsent 重建，支持 TTL/热点参数/容量装填调节。
  - `invalidate_catalog`（CONTINUOUS_WRITE）：批量失效，触发下游 read 的回源重建。
- experimentId=`udqs_plan`，groupId=`default`
  - `lookup_plan`（CONTINUOUS_READ）：查询路由/执行计划缓存。
  - `refresh_plan`（CONTINUOUS_WRITE）：批量刷新计划，模拟统计变化。

### 四、实施步骤（前端 / 控制面统一）

1. **基线运行**
   - 在前端选择 `metadata_cache/default/read_catalog`，`loadShape`: constant 2000 qps, concurrency 64, duration 600。
   - `dataRequest` 示例：
     ```json
     {
       "keySpaceSize": 100000,
       "hotKeyRatio": 0.7,
       "hotKeyCount": 200
     }
     ```
   - `overrides` 留空（沿用 Nacos 基线）；记录命中率、P99、CPU、GC。
2. **调参 1：容量与装填**
   - 在前端 `overrides` 中把 `initialCapacity` 提到 200000、`loadFactor` 0.5，观察扩容次数与 CPU；再降到 16384、`loadFactor` 0.9，看是否出现频繁扩容和尾延迟。
   - 结论：记录“容量/装填”与吞吐、P99、GC 的关系，确定上线预留比例。
3. **调参 2：热点与锁竞争**
   - 提高 `hotKeyRatio` 到 0.95、`hotKeyCount` 10；保留 computeCost 400us，查看 async-profiler 火焰图中 `Node.casTabAt`、`synchronized` 热点。
   - 若出现竞争，将 `computeIfAbsent` 替换为 “double-check + putIfAbsent” 的实现（需要在代码中暴露开关），再跑一轮并对比。
4. **调参 3：失效风暴**
   - 触发 `invalidate_catalog`：`loadShape` constant 200 qps，`dataRequest.invalidateRange=500`，`overrides.burstIntervalMillis=5000`。
   - 观察 read 流量下 map 是否出现全表重建导致的抖动；尝试将 `maxEntries` 限制为 5 万并开启“分批失效 + 延迟重建”策略（如果代码支持），对比 tail 延迟。
5. **UDQS 计划缓存复现**
   - 跑 `udqs_plan/default/lookup_plan`：`qps=3000`，`concurrency=128`，`dataRequest.ttlSeconds=10`。
   - 并行启动 `refresh_plan`：`qps=50`，`refreshBatch=500`；通过 Nacos 将 `computeCostMicros` 提到 800，观察查询计划重建对 CPU、P99 的影响。
6. **记录与回收**
   - 每轮实验后将 `overrides` 写回 Nacos，保持 UI/执行端一致。
   - 保存 Grafana 截图、JFR/async-profiler 火焰图，整理“配置 → 指标”矩阵。

### 五、观察指标与判定

- 读路径：P99/P999、命中率、线程池排队、`ConcurrentHashMap` 冲突/重试（通过 JFR Lock Profiling 或 async-profiler 锁模式）。
- 写/失效：单次失效耗时、导致的 read tail、map size 曲线（监控中可暴露 `map.size()`、`rehashCount`）。
- 内存：老年代占用、对象存活；大 value 下是否导致晋升/碎片。
- CPU：`rehash`、`computeIfAbsent`、序列化/JSON 相关热点。
- 可靠性：失效风暴下是否出现“缓存回源放大”或队列堆积；刷新失败重试策略是否平滑。

- **Grafana/JVM 多层级观测建议**
  - JVM Dashboard：开启 `JMX_PROMETHEUS`（compose 已接入）后，在 Grafana 新增面板观测 `jvm_memory_bytes_used{area="heap"}`、`jvm_gc_pause_seconds_bucket`、`thread_states`。
  - Map 级指标（代码侧暴露 Micrometer Gauge/Counter，再由 Prometheus 抓取）：
    - `concurrent_cache_size{cache=metadata_catalog|udqs_plan}`、`concurrent_cache_max_entries`、`concurrent_cache_value_bytes`
    - `concurrent_cache_hit_total`、`concurrent_cache_miss_total`、`concurrent_cache_invalidate_total`、`concurrent_cache_refresh_total`、`concurrent_cache_evict_total`、`concurrent_cache_rebuild_total`
    - `concurrent_cache_compute_seconds_*`（Timer，用于定位重建慢导致的锁竞争）
  - 入口层：`http_server_requests_seconds_bucket`/`@Timed` 观测 P99；如果是 gRPC/自研 RPC，对应的 `server_request_seconds`。
  - DB/Redis 层：MySQL `global_status` 面板关注 `com_select/sort_scan`，Redis 关注 `keyspace_hits/misses`、`latency_spike`。
  - 采样：并行启动 async-profiler CPU/Lock 模式和 JFR（低采样率，如 100 ms）覆盖整个实验窗口，回放火焰图与 Grafana P99 对齐定位。
- Grafana Dashboard：仓库已放置 `grafana/provisioning/dashboards/concurrent-cache-dashboard.json`（自动加载，Prometheus UID `prom`），涵盖 Map size、Hit/Miss、Rebuild/Refresh/Invalidate/Evict、Compute 平均耗时、Value size/maxEntries 等。

### 六、输出物

- 最终推荐的 `initialCapacity/loadFactor/maxEntries` 组合（按元数据 cache 和 UDQS plan cache 分别给出）。
- 失效/刷新策略建议（全量重建 vs 分批重建 vs 延迟重建），以及 Nacos 上线的默认参数。
- 观察到的热点堆栈（火焰图位置）与优化动作（预设容量、降低 compute 成本、引入分段锁/本地缓存等）。

## wallet_query/default 调参示例

query_snapshot → Heap 膨胀/OOM
wallet_query/default/query_snapshot：

{
  "ttlSeconds": 1800,
  "assetCount": 150,
  "includeHistory": true,
  "historyDays": 14,
  "includeRisk": true,
  "userSegment": "mixed"

长 TTL + 大量资产/历史记录会让缓存和对象同时常驻，老年代只增不减，观察 Heap 曲线或触发 OOM。

query_snapshot → 频繁 Minor GC / STW 抖动

{
  "ttlSeconds": 2,
  "assetCount": 80,
  "includeHistory": true,
  "historyDays": 3,
  "includeRisk": false,
  "userSegment": "hot"
}
TTL 压到 2 秒，每次请求都回源并拼 JSON，短命对象海量产生，Minor GC 次数激增，可在 Grafana/JFR 里看到 P99 与 GC pause 联动。

query_snapshot (VIP) → CPU 热点

{
  "userSegment": "vip",
  "assetCount": 220,
  "includeHistory": true,
  "historyDays": 10,
  "includeRisk": true,
  "ttlSeconds": 15
}
VIP 路径会额外算风险敞口、收益等指标，资产数拉高到 200+ 后 CPU 火焰图会出现 BigDecimal/JSON 序列化热点，适合用 async-profiler 抓栈。

warm_snapshot → Heap/慢 GC 观测

{
  "userRange": {"start": 100000, "count": 90000},
  "batchSize": 1000,
  "ttlSeconds": 1200
}
批量预热 9 万个用户快照，能快速填满缓存并检验后台批任务对堆和线程的影响。

rebuild_ledger → 线程池耗尽/锁竞争

{
  "userRange": {"start": 150000, "count": 5000},
  "batchSize": 100,
  "daysBack": 30,
  "retryMode": "at_least_once"
}
回放 30 天流水、批次小，会让任务排队且在 Redis/MySQL 连接上 BLOCKED，适合复现线程池饱和和 RejectedExecutionException。

publish_bus → Kafka/Metaspace/序列化热点

{
  "topic": "wallet.events",
  "payloadSize": 25600,
  "partitionSkew": 0.98,
  "dynamicLoader": true
}
Payload 提到 25 KB、倾斜 0.98，配合 dynamicLoader（在 service 中启用 ByteBuddy/CGLIB）可观察 CPU、吞吐和 Metaspace 的同步上升，复现实例中的 Kafka 序列化压力。

## 场景 1：用户自选 / 关注币对
### 1. 业务 & MySQL 模型
```sql
favorite_symbol(
  id BIGINT PK,
  user_id BIGINT,
  symbol VARCHAR(32),
  tags VARCHAR(256),
  created_at DATETIME(6),
  INDEX idx_uid(user_id),
  INDEX idx_uid_symbol(user_id, symbol)
)
```
典型行为：
- App/Web 首页 / 自选页：频繁读取 `某个 user 的自选列表`
- 用户偶尔增删自选（写远少于读）
### 2. 访问特征

- 读写比：**100:1 甚至 1000:1**
- 新鲜度：**人眼可见，允许几十毫秒～几秒级延迟**
    - 用户刚加/删一个自选，希望刷新页面立即看到变化；
    - 但偶发 1–2 秒延迟一般可以接受。
- 热点模式：
    - “按 user 分片”：单个用户的自选列表很少；
    - 热点集中在“高活跃用户”和“最近活跃用户”。

### 3. Redis 缓存设计（键粒度）

常见设计：

- Key：`fav:{user_id}` → value: 自选 symbol 列表（JSON / 有序集合 / list）
- 可再配合一个倒排：`fav:by_symbol:{symbol}` 存有哪些用户关注（次要）

### 4. 建议测试的缓存方案 & 观测点

**方案 A：最经典 cache-aside + TTL**

- 读：
    - 先查 Redis `fav:{uid}`，miss → 查 MySQL `idx_uid` → 回写 Redis，附带 TTL（比如 30s/60s）
- 写：
    - Update MySQL（插入/删除 favorite_symbol）；
    - **粗暴做法**：直接 `DEL fav:{uid}`，让下一次读重新从 DB 回源；
    - 不做精确更新，简化逻辑。
测试维度：
- TTL = 5s / 30s / 5min / 不设 TTL，仅依赖显式 DEL
- 观测：
    - 自选读取命中率，DB QPS 降低幅度；
    - 写之后在 TTL 未过期前，读到旧数据的概率（数据新鲜度）；
    - Redis 内存使用 vs TTL 长度；
    - 极端压力下 MySQL 是否成为瓶颈。
**方案 B：写穿（write-through）+ 不设 TTL，只做显式更新**
- 写：
    - 事务内或紧跟 DB 写，将 Redis 中 `fav:{uid}` 做增删操作保持同步；
- 读：
    - Redis 为准，不设 TTL。
测试重点：
- 在高并发“加/删自选”的情况下：
    - 读写一致性（新鲜度几乎实时）
    - 代码复杂度 & 容错（写 DB 成功、写 Redis 失败怎么办？）
    - 对 Redis 命令类型的开销（频繁修改 vs 重建）

### 5. 代码与操作映射

场景 1 已落地在 `favorite/default` 实验组中，提供下列 operation：

| operationId | 类型 | 说明 | 主要 payload |
|-------------|------|------|--------------|
| `read_cache_aside` | CONTINUOUS_READ | 先查 Redis `fav:{userId}`，miss 时回源 MySQL 并写入缓存（可通过 `ttlSeconds` 覆盖默认 TTL） | `userId`(long)、`ttlSeconds`(可选) |
| `add_favorite` | WRITE | 向 MySQL 写入（upsert）并 DEL 缓存，模拟用户新增自选 | `userId`、`symbol`、`tags`(可选) |
| `remove_favorite` | WRITE | 从 MySQL 删除记录并 DEL 缓存，模拟用户移除自选 | `userId`、`symbol` |
| `warm_cache` | ONCE | 只做回源 + 写缓存，不向调用方返回结果，用于批量预热 | `userId`、`ttlSeconds`(可选) |

控制面下发示例：

```json
{
  "experimentId": "favorite",
  "groupId": "default",
  "operationId": "read_cache_aside",
  "loadShape": {
    "type": "hot_key",
    "qps": 2000,
    "concurrency": 64,
    "params": {
      "hotKeyRatio": 0.8,
      "hotKeyCount": 50,
      "keySpaceSize": 100000
    }
  },
  "dataRequest": {
    "userId": 123456,
    "ttlSeconds": 30
  }
}
```

实验过程中可通过 `add_favorite/remove_favorite` 动态写入数据，再用 `read_cache_aside` 压测命中率、新鲜度与 DB 回源情况。

## 场景 2：用户资产总览 / 估值（account + PnL 汇总）

---

### 1. 业务 & MySQL 模型

表/视图可能类似：

```sql
user_balance(
  user_id BIGINT,
  asset   VARCHAR(32),
  free    DECIMAL(32,16),
  locked  DECIMAL(32,16),
  updated_at DATETIME(6),
  PRIMARY KEY (user_id, asset)
)

-- 也可能有一个预聚合的 user_asset_snapshot / user_portfolio

```

业务行为：

- 前端“资产总览页 / 钱包页 / Dashboard”会频繁读取：
    - 每种币的资产余额
    - 折合 USDT、BTC 的估值、总资产、总收益等
- 资产变动来源：
    - 充值/提现
    - 现货/合约交易
    - 手动划转、收益发放等

### 2. 访问特征

- 读写比：**读较多但写也不少**（特别是活跃交易用户）
- 新鲜度要求：
    - 对大部分用户：**秒级新鲜度**即可；
    - 对交易中的用户：希望尽量接近实时。
- 一致性敏感度：
    - 金额类数据 → 用户容忍度比自选低得多；
    - 显示“可用余额”和“总资产”不一致会引发困惑甚至投诉。

### 3. Redis 缓存设计

- Key：`balance:{user_id}` → value: 所有资产及估值的 JSON
    - 或 `balance:asset:{user_id}:{asset}` 拆成多 key
- 可以配合本地缓存（in-process）做二级 cache。

### 4. 建议测试的缓存方案 & 观测点

为了测试**“一致性要求高 + 写量也不低”**场景，建议测：

**方案 A：短 TTL + 写后删除**

- 读：cache-aside + TTL（例如 2s 或 5s）
- 写：每次 balance 变化，写 DB 后 `DEL balance:{uid}`
- 特点：
    - 常见折中方案：
        
        基本保证 1–5 秒内资产是新的；
        
        偶发高并发写时有短暂不一致。
        

测试内容：

- 观察在高频交易用户下：
    - Redis 命中率 vs MySQL QPS；
    - 用户可见“余额跳一下”的频率（可用模拟日志）；
    - Redis 的 big value 更新开销 vs 频繁分散 key 更新的开销。

**方案 B：写穿 + 异步校准**

- 写：
    - 交易撮合 / 资金服务写 DB 的同时同步更新 Redis；
    - 关键路径保证 Redis 立即新鲜；
- 后台起一个“资产对账任务”，定期从 DB 拉全量校准 Redis；

测试内容：

- 人为制造局部故障（写 Redis 失败 / 延迟）：
    - 看资产显示的错误率、恢复速度；
    - 对 MySQL 回源的压力。
  
  ## 场景 3：全站市场榜单/快照（Top Gainers / 热门币种列表）

### 1. 业务 & MySQL 模型

你的数据平台可能会将“交易所 + 链上指标”等写入 MySQL 中的汇总表，例如：

```sql
symbol_metric_snapshot(
  symbol        VARCHAR(32) PK,
  price         DECIMAL,
  price_change_24h DECIMAL,
  volume_24h    DECIMAL,
  open_interest DECIMAL,
  updated_at    DATETIME(6)
)
-- 或者有一张 top_list 表，每 N 秒产出 new snapshot
```

### 2. 访问特征

- 面向所有用户，典型是：
    - “首页热门币种榜单”
    - “涨幅榜 / 跌幅榜 / 成交额榜”
- 读写比：**读非常多，写由后台任务定期写入（比如每 5s / 10s / 60s）**
- 新鲜度：
    - 对榜单类：**秒级 or 10 秒级** 就足够
    - 不需要每条请求都查 DB 再算排序。

### 3. Redis 缓存设计

- Key：`top:gainers:1m`, `top:gainers:24h`, `top:volume:24h`
- Value：
    - 可以用 ZSET（score = 涨幅/成交量）
    - 也可以直接存 JSON 列表（排行榜前 100）

### 4. 建议测试的缓存方案 & 观测点

**方案 A：纯缓存视为源（Redis 为主，周期 DB 写）**

- 后台任务：
    - 从行情/流处理系统计算出最新榜单 → 直接写 Redis；
    - 再异步写 MySQL 作为归档；
- API 查询：
    - 完全从 Redis 拿榜单，不读 MySQL。

测试内容：

- 只对 Redis 压测：
    - key 设计对热点的影响（一个榜单 vs 分片榜单）
    - 不同 Redis 数据结构（ZSET vs JSON）带来的 CPU/内存差异
    - 观察榜单刷新频率 vs 客户端感知新鲜度

**方案 B：MySQL 为主，Redis 做 query 结果缓存**

- 排名查询（复杂 SQL）写在 MySQL 上：
    - `SELECT symbol, ... FROM symbol_metric_snapshot ORDER BY price_change_24h DESC LIMIT 100;`
- 读请求高峰：cache-aside 把查询结果缓存到 Redis（TTL 10s/30s）。

测试内容：

- 在高并发榜单访问下对比：
    - 没有缓存 → MySQL CPU、执行时间、锁等待
    - 有缓存 → MySQL QPS/耗时下降幅度，Redis hit ratio
    - TTL 长短对“新鲜度 vs MySQL 压力”的折中效果

这个场景非常适合用来观察：**复杂查询结果缓存 vs 直接打 MySQL** 对底层资源的影响。
## 场景 4：K 线 / 历史聚合指标（按时间窗口）

### 1. 业务 & MySQL 模型

表可能像：

```sql
kline_1m(
  symbol       VARCHAR(32),
  open_time    DATETIME,
  open_price   DECIMAL,
  close_price  DECIMAL,
  high_price   DECIMAL,
  low_price    DECIMAL,
  volume       DECIMAL,
  PRIMARY KEY (symbol, open_time)
)

```

业务：

- 前端 K 线图 / 策略分析需要读取过去 N 条 K 线，
- 时间分辨率可用 1m / 5m / 1h / 1d 等。

### 2. 访问特征

- 写入：后台流处理 / 批作业 按固定周期写入新 K 线；
- 读取：用户打开图表时高频访问；
- 新鲜度：
    - K 线时间粒度已经是离散窗口，
    - N 秒级延迟通常完全可接受（只要不漏最近几个窗口）。

### 3. Redis 缓存设计

- Key：`kline:{symbol}:{interval}`
- Value：最近 N 条 K 线列表（可能几十 ～ 几百条）

### 4. 建议测试的缓存方案 & 观测点

**方案 A：分段缓存 + 只刷新最近区间**

- 大部分历史数据在 MySQL，只缓存最近比如 500 条；
- 当新 K 线生成时：
    - Append 到 Redis 的 List / Sorted set；
    - 定时淘汰过旧的 K 线条；
- 读不命中历史段时，落到 MySQL 查询。

测试维度：

- N（缓存条数） = 100 / 500 / 2000 的对比：
    - Redis 内存 vs MySQL 压力 vs 磁盘 IO
- 大量用户反复拖动/缩放 K 线，对：
    - Redis 命中率
    - MySQL query pattern（是否大量扫历史）的影响

**方案 B：只缓存“最近一屏”**

- API 接口直接规范：只提供“最近 N 条”；
- Redis 全权负责这个 N 条数据，仅做覆盖更新，不回 MySQL。

测试的重点是：当你**约束业务接口**后，缓存可以极简化，对 MySQL 的依赖大幅降低。

### 5. Kafka 写入实验映射

K 线生成端也可以直接通过 `load-executor` 压测 Kafka topic。本仓库新增了 `kafka_kline/market-data` 实验组，提供 `produce_kline` operation，负责按照给定 QPS + symbol 权重向 `binance.kline`（或自定义 topic）持续写入样例 Payload：

```json
{
  "experimentId": "kafka_kline",
  "groupId": "market-data",
  "operationId": "produce_kline",
  "loadShape": {
    "type": "constant",
    "qps": 500,
    "concurrency": 64,
    "durationSeconds": 600
  },
  "dataRequest": {
    "topic": "binance.kline",
    "interval": "1m",
    "exchange": "binance",
    "symbols": "BTCUSDT:80,ETHUSDT:15,SOLUSDT:5"
  }
}
```

- `symbols` 支持多种形式：字符串 `BTCUSDT:80,ETHUSDT:20`、JSON 数组 `[{"symbol":"BTCUSDT","weight":90,"basePrice":93000}]` 或 `{"BTCUSDT":90,"ETHUSDT":10}`；通过权重就能制造热点（某个 token QPS 极高，其他很低）。
- 可继续在 `loadShape.params` 中填 `hotKey` 参数，让特定 symbol 成为热点，配合 `symbols` 权重调整写入倾斜。
- 生成的消息结构与 Binance WebSocket K 线类似，包含 `eventTime/exchange/interval/kline` 字段，并以 `symbol` 作为 Kafka key，便于下游按分区消费。

> Kafka 连接优化：`application.yml` 中的 `spring.kafka.bootstrap-servers` 等已经改成支持环境变量的形式，例如 `KAFKA_BOOTSTRAP_SERVERS=kafka:29092` 用于容器内访问，或在本机运行时保持 `localhost:9092`。如需调节压缩、acks、linger 等参数，也可以通过 `KAFKA_PRODUCER_*` 变量覆盖。
